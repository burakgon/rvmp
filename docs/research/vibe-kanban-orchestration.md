# Vibe Kanban: task/worktree/process-supervision orchestration (research)

Date: 2026-07-19 · Source: `github.com/BloopAI/vibe-kanban` (Apache-2.0, Bloop AI) · Commit studied: `4deb7eca8f381f7cbc1f9d15515a9ab8f8009053` (v0.1.44, 2026-04-24) · Clone kept at `/Users/burakgon/.claude/jobs/b3095dd6/tmp/vibe-kanban` (shallow, depth 50). All `file:line` refs relative to that repo root at that SHA. Stack: Rust/axum/SQLite (sqlx) backend in `crates/`, React frontend in `packages/`.

**Headline: VK at its final SHA is NOT a queue-driven orchestrator — it is a supervision engine. The product was sunset in April 2026 (README banner #3388; local kanban routes reduced to an export page #3387) after pivoting task tracking to a cloud issue tracker; the automatic kanban transitions (todo→inprogress→inreview) that made VK famous are gone from this code, the `tasks` FK on attempts was dropped (migration `20260217120312_remove_task_fk_from_workspaces.sql`), and there is NO worker limit, NO dispatch queue, NO heartbeat, and NO circuit breaker anywhere (zero semaphores/concurrency caps in `crates/`). What survives — and is battle-tested across ~3,390 PRs — is exactly our R2-R4 + worktree substrate: a 4-level Workspace→Session→ExecutionProcess→Turn model with soft-drop history rewind, a worktree manager with per-path creation locks and 3-tier orphan cleanup, a process supervisor with a 30 s spawn timeout / 250 ms exit poll / SIGINT→SIGTERM→SIGKILL 2 s ladder, squash-merge with task-branch ref reset (their squash-detection fix), 60 s PR polling → auto-archive, and same-session follow-up/review rounds via stored `agent_session_id`. Notably, their "completion = exit code" model already leaks toward declared-done: Codex/Opencode/Gemini/Qwen run as non-exiting protocol servers whose executors *signal* turn completion (`ExecutorExitSignal`), vindicating codegent's declared-completion stance.**

## 1. Data model: task → workspace (attempt) → session → execution process → turn

- `TaskStatus = { Todo, InProgress, InReview, Done, Cancelled }` still exists (`crates/db/src/models/task.rs:14-21`) but is vestigial at this SHA: no code in server/services mutates it; tasks graduated to remote Issues with free-form `ProjectStatus` rows (name/color/sort_order/hidden, `crates/api-types/src/project_status.rs:9-17`) and f64 `sort_order` on issues (`crates/api-types/src/issue.rs:33`) — Linear-style manual columns, no automatic transitions in the remote model either.
- **Workspace = attempt** (renamed from `task_attempts` in migration `20251216142123_refactor_task_attempts_to_workspaces_sessions.sql`): `{id, task_id(Option), container_ref(worktree dir path), branch, setup_completed_at, archived, pinned, name, worktree_deleted}` (`crates/db/src/models/workspace.rs:42-54`). Name auto-derived from first user prompt, truncated to 60 chars at word boundary (`WORKSPACE_NAME_MAX_LEN`, workspace.rs:10, 484-496).
- **Session** = one conversation with one executor inside a workspace: `{id, workspace_id, name, executor, agent_working_dir}` (`crates/db/src/models/session.rs:22-31`). Executor is pinned per session — follow-up with a different executor is rejected (`ExecutorMismatch`, session.rs:18-19; enforced in `follow_up`, `crates/server/src/routes/sessions/mod.rs:148-162`).
- **ExecutionProcess** = one OS process run: `run_reason ∈ {SetupScript, CleanupScript, ArchiveScript, CodingAgent, DevServer}`, `status ∈ {Running, Completed, Failed, Killed}`, `exit_code`, `started_at/completed_at`, `dropped` (soft-delete for history rewind — hidden from logs/timeline, still listed in Processes tab) and the full serialized `executor_action` JSON (`crates/db/src/models/execution_process.rs:43-78`). Per-repo git state per process: `execution_process_repo_states` with `before_head_commit`/`after_head_commit`/`merge_commit` captured at start/end (`crates/services/src/services/container.rs:1156-1166`; `crates/local-deployment/src/container.rs:325-341`).
- **CodingAgentTurn** = one prompt→response round: `{execution_process_id, agent_session_id, agent_message_id, prompt, summary, seen}` (`crates/db/src/models/coding_agent_turn.rs:7-18`). `agent_session_id`/`agent_message_id` arrive as `LogMsg::SessionId`/`MessageId` sentinels parsed out of the agent's stream and stored mid-run (`crates/services/src/services/execution_process.rs:309-343`). `summary` = last assistant message, ≤ 4096 chars (local-deployment/container.rs:897-925), reused as the auto-commit message.
- **Action chain**: `ExecutorAction { typ, next_action: Option<Box<ExecutorAction>> }` — a linked list serialized into the process row; on success the exit monitor starts `next_action` as a new ExecutionProcess, inferring its run_reason (script→script = SetupScript; agent→script = CleanupScript; anything→agent = CodingAgent) (`crates/executors/src/actions/mod.rs:35-72`; container.rs:1358-1391). Standard chain: setup script(s) → coding agent → cleanup script. Setup scripts can run parallel (independent processes) or sequential (chained via next_action) per-repo flag (container.rs:1095-1128).
- **What survives restart: everything is SQLite + JSONL files; processes do not.** At startup `cleanup_orphan_executions` marks every `running` row **Failed** with `exit_code = NULL` and captures after-head commits (container.rs:273-326) — no resume of in-flight processes, ever. On graceful shutdown `kill_all_running_processes` marks them Killed (local-deployment/container.rs:1603-1633; `crates/server/src/main.rs:236-242`). Worktrees are recreated on demand from DB state (`ensure_container_exists`, see §2).
- **Retry ≠ new attempt.** Retry rewinds the SAME workspace: `follow_up` with `retry_process_id` → `reset_session_to_process` = git-reset each repo worktree to that process's `before_head_commit` (fallback: previous process's after-commit), `try_stop` everything, soft-`drop` all processes at/after the boundary (`crates/services/src/services/container.rs:632-705`; sessions/mod.rs:169-176), then spawn a follow-up with `reset_to_message_id` so the agent session also rewinds (§6). Defaults: `perform_git_reset=true`, `force_when_dirty=false` (sessions/mod.rs:170-171). The old worktree is never discarded on retry. A truly fresh attempt = a new Workspace (new branch, new worktree, new session). Frontend greys out processes after the retry target (`packages/web-core/src/features/workspace-chat/model/contexts/RetryUiContext.tsx:29-38`).

## 2. Worktree lifecycle

- **Layout**: one workspace dir containing one worktree **per repo** (multi-repo): `{base}/{4-hex-short-uuid}-{slug}/{repo_name}` where slug = `git_branch_id(name)` = lowercase, non-alnum→`-`, max 16 chars (`crates/utils/src/text.rs:4-18`), short_uuid = first 4 hex chars of the workspace UUID (text.rs:20-24), dir name built at `crates/local-deployment/src/container.rs:852-855`. Base dir: macOS `$TMPDIR/vibe-kanban/worktrees`, Linux `/var/tmp/vibe-kanban/worktrees` (deliberately not `/tmp` to avoid RAM tmpfs), `-dev` suffix in debug (`crates/utils/src/path.rs:108-125`; `crates/worktree-manager/src/worktree_manager.rs:511-523`). Custom override always nests an app-owned `.vibe-kanban-workspaces/` subdir "so orphan cleanup never touches user folders" (worktree_manager.rs:512-517).
- **Branch naming**: `{prefix}/{short_uuid}-{slug}` with default prefix `"vk"` (`crates/services/src/services/config/versions/v8.rs:12-14`; `git_branch_from_workspace`, container.rs:786-795). Branch name is identical across all repos of a workspace. Base ref = per-repo `target_branch` chosen at creation (stored on `workspace_repos`); branch created from it via `create_branch` then `git worktree add` (CLI, to inherit sparse-checkout semantics) (`worktree_manager.rs:59-84, 314-317`).
- **Creation is serialized per worktree path** via a global `HashMap<String, Arc<tokio::sync::Mutex<()>>>` (worktree_manager.rs:16-17, 95-105). `ensure_worktree_exists` = check filesystem + git metadata (`worktrees/*/gitdir` matching, canonicalized, macOS `/private` alias-normalized) → if broken, comprehensive cleanup (git worktree remove --force → force-delete metadata dir → rm -rf path → `git worktree prune`) and recreate with one retry after metadata cleanup (worktree_manager.rs:88-183, 229-265, 302-365). Multi-repo creation rolls back all already-created worktrees on any failure (`crates/workspace-manager/src/workspace_manager.rs:290-371`). Cold-restart recreation: if branch exists → reattach worktree; if branch missing → recreate from target branch (workspace_manager.rs:374-425).
- **Setup/env copying** at create + ensure: (1) per-repo `copy_files` globs copied from origin repo (30 s timeout, skip same-size existing) (local-deployment/container.rs:946-989, 1582-1601); (2) workspace-level `CLAUDE.md`/`AGENTS.md` generated with `@{repo_name}/CLAUDE.md` import lines when repos have them (container.rs:991-1045); (3) setup scripts as bash ExecutionProcesses (§1). Env injected into every process: `VK_WORKSPACE_ID`, `VK_WORKSPACE_BRANCH` (local-deployment/container.rs:1365-1367).
- **Cleanup/archival triggers** (worktrees are cattle; DB rows are pets):
  - Periodic sweep every **30 min** deletes worktrees of expired workspaces: **72 h** idle for active, **1 h** for archived, measured from max(workspace.updated_at, last process completed_at), only when no non-dev-server process is running; sets `worktree_deleted=TRUE`, row survives (`find_expired_for_cleanup`, workspace.rs:252-309; sweep spawn local-deployment/container.rs:299-320). `DISABLE_WORKTREE_CLEANUP` env var disables both sweeps (container.rs:276-282).
  - Orphan scan at startup: any dir under the base dir whose path is not a `container_ref` in the DB is removed (per-subdir `.git`-file check → full worktree cleanup) (workspace_manager.rs:538-652).
  - `touch()` on access (e.g. opening in editor) refreshes updated_at with a **2 min** debounce (`WORKSPACE_TOUCH_DEBOUNCE`, local-deployment/container.rs:66, 1152-1183).
  - Workspace deletion: stop processes → cleanup worktrees → delete branches (optional) → remove per-session JSONL logs (workspace_manager.rs:217-288).
- **Archive is a first-class operation** (their merge→archive): `archive_workspace` = set `archived` flag → kill running dev servers → run optional per-repo `archive_script` as an ArchiveScript process (container.rs:480-561). Archived ≠ deleted: the accelerated 1 h window deletes the worktree later. Starting any non-archive execution auto-UN-archives (container.rs:1184-1192).
- **Merge (direct)**: squash-only. Guard rails: refuse if an open PR exists for the repo; refuse if target branch is remote-only; **refuse if the task branch is behind base at all** (`task_behind > 0` → `BranchesDiverged` — user must rebase first) (`crates/server/src/routes/workspaces/git.rs:195-232`; `crates/git/src/lib.rs:575-593`). Commit message = `"{workspace_label} (vibe-kanban {issue-or-workspace-id})"` (git.rs:222-224). Two paths: base branch checked out somewhere → CLI `merge --squash` in that checkout (staged-changes safety check first); otherwise pure in-memory libgit2 squash: `merge_commits` with `fail_on_conflict(true)`, single parent = base commit, no working-tree contact (lib.rs:594-671, 1082-1122). **After merging, the task branch ref is reset to the squash commit** ("so follow-up work can continue from the merged state without conflicts", lib.rs:658-668) — this simultaneously kills the classic squash-merge false-negative: ahead/behind counts vs base become 0/0 immediately. Then: record `DirectMerge` row → **auto-archive unless pinned** (git.rs:234-254).
- **Merge detection today**: no patch-id, no commit scanning. Direct merges are *recorded events* (merges table, `crates/db/src/models/merge.rs:74-106`); PR merges are *polled state* — `PrMonitorService` every **60 s** asks the host (`gh pr view --json`) for each PR in local `pull_requests` with status Open; on Merged: store `merged_at` + `merge_commit_sha`, and archive the workspace when its open-PR count hits 0 and it isn't pinned (`crates/services/src/services/pr_monitor.rs:59-211`). Merge state is workspace-scoped truth, not git-scoped inference.
- **Rebase/conflicts**: `rebase_branch` endpoint supports changing base branch; conflict → structured `MergeConflicts {message, op, conflicted_files, target_branch}` response, worktree left in conflicted state for the user/agent, plus `rebase/continue` and `conflicts/abort` endpoints; `BranchStatus` streams `is_rebase_in_progress`, `conflicted_files`, `conflict_op` (git.rs:694-840, 411-429). Branch rename cascades: **child workspaces targeting the old branch get retargeted** (`update_target_branch_for_children_of_workspace`, git.rs:664-678) — their stacked-attempts cascade.

## 3. Process supervision

- **Spawn**: `executor_action.spawn(current_dir, approvals, env)` under a **30 s timeout** ("Timeout: process took more than 30 seconds to start", local-deployment/container.rs:1369-1379); spawned as a **process group** (`command_group::AsyncGroupChild`, `group_spawn_no_window`) with `kill_on_drop(true)` and piped stdio (e.g. claude.rs:629-651). Spawn failure → process marked Failed, stderr log line `"Failed to start execution: {e}"`, and a structured `SetupRequired` error entry if the executable is missing (container.rs:1228-1295).
- **Exit detection, two channels raced with `tokio::select!`** (`spawn_exit_monitor`, local-deployment/container.rs:479-527):
  1. OS exit watcher: polls `child.try_wait()` every **250 ms** (container.rs:815-850).
  2. **`ExecutorExitSignal`** — a oneshot from the executor for agents whose process does NOT exit at turn end (comment at container.rs:505-507). Used by Codex, Opencode, and the ACP harness (Gemini/Qwen) (`grep exit_signal: Some` → codex.rs:758, opencode.rs, acp/harness.rs). On signal: kill the process group, then synthesize exit status from the signal payload (`ExecutorExitResult::Success→0 / Failure→1`; closed channel = success) (container.rs:508-527, 1121-1132, 1635-1646).
- **Status mapping**: exit code 0 → `Completed`, non-zero → `Failed` (code stored, `unwrap_or(-1)`), wait-error → Failed with NULL code (container.rs:529-540). **Completion dedup**: `update_completion` is skipped if the row is already Killed/Completed (`was_stopped` guard, container.rs:542-547; execution_process.rs:418-428) — DB status is a write-once terminal latch; a user Stop racing a natural exit can't double-fire.
- **Kill ladder** (`utils::process::kill_process_group`, `crates/utils/src/process.rs:5-32`): SIGINT → sleep 2 s → SIGTERM → sleep 2 s → SIGKILL, each sent to the **process group** via `killpg` with the pgid captured at spawn ("works even after the group leader has exited"); ESRCH short-circuits; then `child.kill()` + `wait()`. `stop_execution` first fires the executor's `CancellationToken` (protocol-level graceful cancel, e.g. Claude control-protocol interrupt) and waits up to **5 s** for the exit monitor before the ladder (local-deployment/container.rs:1405-1457).
- **Zombie/orphan reaping**: after the executor is done, any children still in the group (MCP servers etc.) get a direct `start_kill()` (SIGKILL) and the child entry is dropped (container.rs:804-811; commit `71a25f4` "reap execution process groups on natural exit" #3318). Startup orphan handling is DB-side (§1) — a VK crash leaves real OS orphans to die with their group; there is no PID-file adoption.
- **Post-exit pipeline** (ordered, in the exit monitor, container.rs:549-812): update turn summary from last assistant message → if success (or cleanup finished): **auto-commit** any uncommitted changes with the turn summary as message (per-repo, continues on per-repo failure; commit failure treated as "changes made" to be safe) → start `next_action` only if the agent actually changed something (`changes_committed || has_commits_from_execution` — before/after HEAD comparison; skips the cleanup script otherwise) → else finalize → consume queued follow-up (only on success; discarded on Failed/Killed) → mark turn **unseen** (the UI "needs review" dot) when a CodingAgent process finalizes with no chained/queued follow-up → fire analytics (`task_attempt_finished` with success/exit_code) → capture after-HEAD OIDs → flush MsgStore→DB with **5 s** timeout → reap group.
- **Log capture/storage**: stdout/stderr byte streams → `LogMsg::Stdout/Stderr` chunks → in-memory `MsgStore` per execution (live WS streaming + normalization input) AND appended as JSONL to `{asset_dir}/…/{session_id}/processes/{process_id}.jsonl` (`spawn_stream_raw_logs_to_storage`, execution_process.rs:256-346; path scheme `crates/utils/src/execution_logs.rs:10-22`). Normalized conversation view = per-executor `normalize_logs` parsing the raw stream into `NormalizedEntry` JSON-patches; for historical processes the raw JSONL is replayed through the same normalizer on demand, with consecutive same-path patch dedup (container.rs:830-1045). Logs are deleted with the workspace, not the process.

## 4. Queue/concurrency: there is none

- **No worker limit, no pending queue, no scheduler loop.** Zero hits for semaphore/concurrency caps in `crates/`. `start_workspace` (HTTP or MCP `start_workspace` tool) creates worktree + session + processes immediately (container.rs:1047-1131; `crates/mcp/src/task_server/tools/task_attempts.rs:94-200`). Concurrency is bounded only by the human/orchestrating agent.
- The only per-resource serialization: worktree-path creation mutex (§2), and "one non-dev-server process per workspace" guards on review/setup entry points (`ProcessAlreadyRunning`, `crates/server/src/routes/sessions/review.rs:56-62`, sessions/mod.rs:269-275). Follow-ups during a run are **queued, not parallel**: exactly one pending message per session, replace-on-requeue, in-memory DashMap + Scratch draft persistence, consumed at finalization, discarded if the run Failed/was Killed (`crates/services/src/services/queued_message.rs:31-90`; consumption container.rs:619-673; routes `sessions/{id}/queue` GET/POST/DELETE, `crates/server/src/routes/sessions/queue.rs`).
- Implication: VK contributes nothing to R1 (queue→start-topmost). Orca's coordinator (2 s poll, max 4 concurrent, 1 dispatch/tick — see orca-agent-state.md §6) remains the only real R1 prior art in our set.

## 5. Completion/failure detection & failure UX

- Success = process exit 0 (or executor Success signal). No timeouts on run duration, no output-based inference, no heartbeat. Killed (user stop / archive / shutdown) is terminal and suppresses notifications (container.rs:239-242).
- `finalize_task` fires an OS notification "Workspace Complete: {name}" with ✅/❌ + branch + executor (container.rs:238-270). Board-level state is derived SQL, not stored: `is_running` = any running setup/cleanup/codingagent process; `is_errored` = latest such process ∈ {failed, killed} (workspace.rs:518-536).
- Failure UX: conversation shows the error entry (stderr tail / SetupRequired card); the chat box becomes a follow-up prompt against the same session; retry = pick any previous process → rewind (§1) — the UI greys everything after the retry target (RetryUiContext.tsx:29-38). No automatic retries anywhere. Queued messages die with the failure (container.rs:662-670).
- **Waiting-for-input analog = approvals**: executors bridge permission requests into a central `Approvals` service — pending entry + WS patch broadcast + OS notification, default **timeout 10 h** (`APPROVAL_TIMEOUT_SECONDS = 36000`, `crates/utils/src/approvals.rs:6`), timeout → `ApprovalOutcome::TimedOut` denial (`crates/services/src/services/approvals.rs:86-125`). For Claude this rides the stream-json control protocol (`can_use_tool` / hook callbacks with `permissionDecision: ask`), not a PTY prompt (claude/client.rs:349-380).
- **The Stop-hook commit gate** (their closest thing to completion validation): VK registers a Claude `Stop` hook; on stop, if any repo has uncommitted changes the stop is **blocked** with `{"decision":"block","reason": commit_reminder_prompt + git status}` — default prompt "There are uncommitted changes. Please stage and commit them now with a descriptive commit message."; `stop_hook_active` guards infinite loops; config-toggleable (claude.rs:210-218; claude/client.rs:329-347; config/mod.rs:23).

## 6. Follow-up rounds (their review-comment analog)

- Every CodingAgent process stores the executor-native `agent_session_id` + `agent_message_id` on its turn (§1). A follow-up looks up the latest non-dropped turn with a session id (`find_latest_session_info`, coding_agent_turn.rs:36-57) and spawns a **new OS process** that *resumes the same agent session*: Claude `--resume <session_id>`, plus `--resume-session-at <message_uuid>` when rewinding ("truncates Claude's conversation history to the specified message", claude.rs:360-377). No session found → falls back to a fresh `CodingAgentInitialRequest` in the same worktree (sessions/mod.rs:191-208). So: conversation continuity across processes = executor-native session resume; VK never rebuilds context by re-prompting history.
- **Review round**: `POST /sessions/{id}/review` builds a prompt "Please review the code changes… Repository: X… Review all changes from base commit {fork_point} to HEAD. Use `git diff {base}..HEAD`…" (fork-point computed per repo), optionally + user text, and runs it as a `ReviewRequest` that resumes the session when one exists (review.rs:42-139; prompt builder executors/mod.rs:369-396).
- **PR-comment round**: PR comments are fetched via `gh pr view --json comments` normalized to `UnifiedPrComment` (pr.rs:390+; `crates/git-host/src/github/cli.rs:516`); the frontend feeds them into a follow-up prompt. Auto-PR-description: after PR creation a follow-up run with a templated prompt (`{pr_number}`/`{pr_url}` placeholders) updates the description (pr.rs:96-186).
- While a process runs, "send" becomes "queue" (§4): one replaceable pending message consumed at finalization — VK's answer to mid-run comment pileup.

## 7. PR flow

- Create: push branch via git CLI (user's own git auth; `PushRejected` → explicit ForcePushRequired error path with a separate force endpoint, git.rs:268-361) → provider from remote URL (GitHub via **`gh` CLI** — `gh pr create`/`view --json`/`pr list`, exit code 4 = auth failure; Azure DevOps also supported; token storage fully delegated to the provider CLI, `crates/git-host/src/github/cli.rs:144-220`) → `PullRequest` row (Open) + remote sync → auto-open browser (pr.rs:188-388). Existing PRs can be attached to a workspace (pr.rs:390+).
- Track: 60 s poll of every Open PR (§2). Merged → update row (merged_at, merge_commit_sha) → archive workspace when open-PR count = 0 and not pinned (pr_monitor.rs:128-211). Direct merge and PR merge share the same terminal effect: archive, not a status column write — at this SHA "Done" is the archived workspace + completed issue on the remote tracker.

## 8. Multi-agent executor abstraction

- 9 executors: ClaudeCode, Amp, Gemini, Codex, Opencode, CursorAgent, QwenCode, Copilot, Droid, as an `enum_dispatch` enum `CodingAgent` with serde SCREAMING_SNAKE_CASE ids (executors/mod.rs:93-124). Trait `StandardCodingAgentExecutor`: `spawn`, `spawn_follow_up(session_id, reset_to_message_id)`, `spawn_review` (defaults to follow-up-or-spawn), `normalize_logs`, `default_mcp_config_path`, `get_availability_info`, `discover_options` (models/agents streamed as JSON patches), `get_preset_options` (executors/mod.rs:222-300).
- Config layering: `ExecutorConfig {executor, variant, model_id, agent_id, reasoning_id, permission_policy}` + per-executor structs with `CmdOverrides` (base command override, append-prompt suffix) applied at build time (`apply_overrides`; `AppendPrompt.combine_prompt`, executors/mod.rs:338-360). Capabilities enum drives UI: `SessionFork` (Claude/Opencode/Codex/Gemini/Qwen), `SetupHelper` (Codex/Cursor), `ContextUsage` (Claude/Opencode/Codex) (executors/mod.rs:177-201).
- Normalization layer: per-executor log processors parse native output (Claude stream-json incl. partial messages; Amp thread format; ACP) into shared `NormalizedEntry` types + `LogMsg::SessionId/MessageId` sentinels — the DB/UI never see executor-native formats.
- MCP config injection: per-executor knowledge of config path + JSON shape (`mcp_servers` for Codex TOML-ish, `amp.mcpServers`, `mcp` for Opencode, `mcpServers` default) (executors/mod.rs:126-171).
- **Claude specifics**: headless `claude -p` with `--verbose --output-format=stream-json --input-format=stream-json --include-partial-messages --replay-user-messages`; `--permission-mode=bypassPermissions` default; `--disallowedTools=AskUserQuestion` in non-interactive mode; plan/approvals modes switch to `--permission-prompt-tool=stdio` + hook-callback approval routing; prompt delivered as a stream-json user message over stdin via a control-protocol peer (initialize with hooks → set_permission_mode → send_user_message); graceful cancel via CancellationToken → protocol interrupt (claude.rs:148-196, 620-712). Availability probe = mtime of `~/.claude.json` (claude.rs:601-617).
- Codex/Opencode/ACP executors run as resident protocol servers; their executor tasks watch the native event stream and fire the ExecutorExitSignal on turn end (§3) — i.e., VK normalizes "declared done" (JSON event) into a synthetic exit code.

## 9. VK's own MCP server (orchestration surface)

- `vibe-kanban mcp` serves two modes (`crates/mcp/src/task_server/mod.rs:44-100`): **Global** (full tool set: workspaces, repos, remote projects/issues/tags/assignees/relationships, sessions) and **Orchestrator** (trimmed: context + workspaces + sessions, with `list_workspaces`/`delete_workspace` removed — `tools/mod.rs:52-73`) so an agent can itself spawn/steer sub-workspaces.
- Context bootstrap: the MCP server resolves *which workspace it lives in* by calling `GET /api/containers/attempt-context?container_ref={cwd}` (500 ms timeout) — the server matches cwd against `container_ref` prefixes (deepest-path wins, workspace.rs:348-389) and returns workspace + repos + `orchestrator_session_id` (mod.rs:104-152). Env-free, cwd-based identity.
- **There is no `task_complete` tool and no self-report of completion anywhere** — the MCP is for CRUD + spawning; completion stays exit/signal-based (§3, §5).

## 10. Verdict for codegent

### (a) Adopt (with their concrete values)

1. **The 4-level model with per-process git state.** Card→Attempt→(Session)→Process, each process storing its full action payload plus `before/after_head_commit` per repo, is what makes VK's rewind-retry, "did the agent change anything", diff bases, and orphan recovery all cheap. For R2-R4 bookkeeping, record `{before_head, after_head}` on every attempt at minimum. The `dropped` soft-delete flag (never hard-delete history; rewind = mark dropped) is the right shape for our error/recovery actions — resume/restart/discard map onto their follow-up / reset+follow-up(reset_to_message) / new-workspace triad.
2. **Worktree mechanics wholesale**: per-path async creation mutex; `ensure_exists` = validate filesystem + `worktrees/*/gitdir` metadata → comprehensive cleanup (remove --force, metadata rm, path rm, prune) → create with one retry; rollback-all on partial multi-step creation; recreate-on-demand from DB (worktrees as cattle). Their orphan story is 3-tier and all values are provable: startup scan of base dir vs DB `container_ref`s; 30 min expiry sweep; 72 h active / 1 h archived windows; 2 min touch debounce; `DISABLE_WORKTREE_CLEANUP` escape hatch. For cg: our `cg/<id>-<slug>` matches their `vk/{short_uuid}-{slug}`; adopt their 16-char slug cap + 4-char id and the app-owned-subdir rule for custom worktree roots. Auto-archive-on-merge: copy the **pinned** exemption and the archive = flag + kill-dev-servers + optional archive-script + delayed worktree deletion decomposition.
3. **Supervision numbers**: 30 s spawn timeout; 250 ms exit poll; SIGINT→2 s→SIGTERM→2 s→SIGKILL to the **process group** with spawn-time pgid; protocol-level cancel first with 5 s grace; SIGKILL leftover group members after the leader is done (MCP servers are the canonical orphans); 5 s log-flush timeout before teardown; write-once terminal status latch (`was_stopped` guard) as completion dedup at the DB layer; startup rule "running rows → Failed, capture HEAD". JSONL-per-process log files + in-memory store for live streaming, normalization as a replayable pure function of the raw stream.
4. **Merge/PR flow**: squash merge with `fail_on_conflict` in-memory merge (never leaves a conflicted worktree from a merge button); **refuse merge when behind base** (force explicit rebase — cleaner than auto-rebase); **reset the card branch to the squash commit after merging** (kills squash-detection false-negatives AND makes post-merge follow-ups conflict-free); record merges as events, poll PR state at 60 s via `gh` CLI (auth delegated to `gh`, zero token storage); merged → archive cascade gated on open-PR-count = 0 and not-pinned. Child-branch retarget cascade on rename is the pattern for our merge-cascade of stacked cards.
5. **Follow-up rounds via executor-native session resume** (`--resume` + stored `agent_session_id`, captured by parsing the stream, with `--resume-session-at` for rewind), and the **one-replaceable-queued-message-per-session** rule — consumed on success, discarded on failure — is a good, simple semantics for review comments arriving mid-run.
6. **The Stop-hook commit gate** (block agent stop while worktree dirty, with loop guard) — directly compatible with codegent's declared-done: wire the same check into our `task_complete` MCP handler (reject completion while uncommitted changes exist, return the git status as the reason).

### (b) Avoid / conflicts with codegent's model

1. **Exit-code-as-completion.** It forced VK to bolt on `ExecutorExitSignal` for every protocol-resident agent (Codex/Opencode/Gemini/Qwen) and still can't distinguish "process died" from "task done" for interactive TUIs — in codegent's terminal-resident world every agent is the non-exiting case, so their core completion mechanism is structurally inapplicable. Our declared `task_complete` is the generalization they were converging on (JSON event → completion); keep exit/crash strictly as a *failure* signal (their mapping: unexpected exit while card is Running → Failed attempt, never → Done).
2. **No scheduler.** Nothing here helps R1; do not model our queue on VK. Their absence of limits is viable only because a human clicks Start; an autonomous queue needs Orca-style caps (worker-limit, 1-dispatch-per-tick) plus our heartbeat/circuit-breaker.
3. **Supervised-pipe agent driving** — stdin stream-json prompts, control-protocol approvals, log normalization into their own chat UI — is their biggest subsystem (Claude executor alone: 3,284 lines + client/protocol) and exactly what codegent's real-PTY model makes unnecessary. Also skip: auto-commit of agent leftovers (wrong side of our review gate — a dirty worktree should fail `task_complete`, not get silently committed), per-executor availability probing via config-file mtimes, and their remote-sync layer.
4. **Retry semantics caution**: VK's rewind-in-place mutates the shared worktree during reset (git reset + optional force when dirty). With a live PTY attached to that worktree, in-place rewind is hazardous — codegent restarts should prefer fresh attempt (new dispatch id, same card branch) or require the terminal to be dead first.

### (c) §6.1 dispatch-envelope impact

1. **Nothing in VK argues against our envelope; it argues for it.** VK has no dispatch ids and pays for it with special cases (`was_stopped` latch as the only dedup; queued-message discard rules; exit-signal synthesis). Keep card_id + attempt_id + dispatch_id; VK adds the *DB-side* dedup pattern worth copying verbatim: completion writes are conditional on `status = running` (terminal-state latch), so a duplicate `task_complete` or a racing kill can never double-transition — enforce our dedup at the store, not just by envelope comparison.
2. **Heartbeat/circuit-breaker numbers unchanged** (Orca-derived: 5 min heartbeat, 10 min hung-warn, breaker at 3 failed attempts — VK has no counterparts, and its no-auto-retry policy supports tripping the breaker into a human-facing error state rather than silent retries; VK's per-attempt "3 failures" analog simply doesn't exist because retries are always human-initiated).
3. **Add two supervision timeouts VK proved out**: 30 s dispatch-spawn timeout (spawn failure = attempt Failed immediately with a SetupRequired-style structured reason — do not consume a breaker count for missing executables; VK distinguishes `ExecutableNotFound` and renders install guidance) and 5 s graceful-cancel grace before the signal ladder (SIGINT→2 s→SIGTERM→2 s→SIGKILL, pgid-scoped). Persist `before_head_commit` per dispatch as part of the envelope's repo state — it is what makes discard/restart implementable as a pure git reset later.
